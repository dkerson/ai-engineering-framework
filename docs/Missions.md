# Missions

> Toda interacao do usuario deve ser interpretada como uma Mission.

## Mission Types

| Mission Type | Quando usar |
|--------------|-------------|
| Development Mission | Criar ou evoluir funcionalidade |
| Bug Resolution Mission | Corrigir erro ou comportamento quebrado |
| Architecture Mission | Decisao estrutural, arquitetura ou trade-off tecnico |
| Performance Mission | Lentidao, otimizacao, throughput ou experiencia lenta |
| Security Mission | Seguranca, auth, permissao, vulnerabilidade |
| Data Intelligence Mission | SQL, BI, Power BI, indicadores, divergencia |
| Dashboard Mission | Dashboards, relatorios, KPIs e analytics visual |
| Product Evolution Mission | Melhorar produto como um todo |
| Innovation Mission | Criar diferencial, explorar nova proposta ou experiencia |
| Modernization Mission | Sistema velho, UI antiga, experiencia defasada |
| Growth Mission | Vender mais, crescer, ativar, reter, converter |
| Marketing Mission | Landing pages, campanhas, copy, SEO |
| Brand Mission | Marca, tom, identidade, software premium |
| Launch Mission | Preparar produto para lancamento |
| Scalability Mission | Preparar para escala tecnica ou operacional |
| UX Mission | Reduzir friccao, cliques, suporte, confusao |
| UI Mission | Aparencia, visual, layout, design review |
| Accessibility Mission | Acessibilidade, inclusao e usabilidade assistiva |
| Technical Debt Mission | Divida tecnica, refatoracao, manutencao |
| Quality Mission | QA, confiabilidade, testes, regressao |
| Continuous Improvement Mission | Backlog evolutivo apos entrega |

## Mission Score

Toda mission recebe:

- Valor esperado
- Complexidade
- Esforco
- Impacto
- Prazo estimado
- Confianca
- Risco

## Mission Memory

Durante a mission, o framework deve preservar:

- Objetivo
- Decisoes
- Hipoteses
- Roadmap
- Analises
- Validacoes
- Quick wins
- Backlog sugerido

Mission Memory nao substitui Working Context; ela o orienta estrategicamente.
